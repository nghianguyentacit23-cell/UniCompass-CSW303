# Team Operating Agreement and RACI Matrix

## Team Working Agreement

| Area | Agreement |
|---|---|
| Communication Channel | The team uses Messenger or Discord for quick discussion and Google Drive for shared documents. |
| Meeting Rhythm | The team meets at least twice per week: one planning/check-in meeting and one review meeting. |
| Meeting Rule | Members should join on time, report progress honestly, and raise blockers early. |
| Task Management | All tasks are tracked in Trello or Jira. Each task must have an owner, status, and deadline. |
| Documentation Rule | Important decisions, requirements, and weekly progress must be documented clearly. |
| Quality Rule | Work should be reviewed by at least one other member before being considered done. |
| Conflict Handling | If members disagree, the team discusses based on project goals, course requirements, and available time. If needed, the Product Owner makes the final product decision and the Tech Lead makes the final technical decision. |
| Definition of Done | A task is done when the expected output is completed, reviewed, uploaded or committed to the shared location, and updated on the board. |

## Team Roles

| Role | Main Responsibility |
|---|---|
| Product Owner (PO) | Owns product direction, prioritizes work, and makes final product decisions. |
| Scrum Master (SM) | Facilitates teamwork, tracks progress, and helps remove blockers. |
| Business Analyst (BA) | Writes requirements, personas, scenarios, and user stories. |
| UX Designer (UX) | Designs user flows, wireframes, and ensures the product is easy to understand. |
| Tech Lead (TL) | Owns technical direction, development standards, and technical decisions. |
| Developer (Dev) | Implements features, fixes issues, and supports testing. |

## RACI Matrix

| Activity | PO | SM | BA | UX | TL | Dev |
|---|---|---|---|---|---|---|
| Define product vision | A | C | R | C | C | I |
| Draft personas | C | I | A/R | R | C | I |
| Draft scenarios | C | I | A/R | R | C | I |
| Prepare initial backlog | A | C | R | C | C | I |
| Prioritize user stories | A/R | C | C | C | C | I |
| Design user flow | C | I | C | A/R | C | I |
| Create wireframes | C | I | C | A/R | C | I |
| Plan technical solution | C | I | I | I | A/R | C |
| Implement features | I | I | C | C | A | R |
| Track weekly progress | I | A/R | C | I | C | C |
| Review completed work | A | C | R | R | C | C |
| Prepare weekly submission | A | C | R | R | C | I |
| Prepare final presentation | A | R | R | R | C | C |

## RACI Legend

| Label | Meaning |
|---|---|
| R | Responsible: person who directly performs the work. |
| A | Accountable: person who owns the final result and decision. |
| C | Consulted: person who provides input before the work is finalized. |
| I | Informed: person who should be kept updated. |
